## shadcn/ui Setup

This project uses shadcn/ui as its component foundation. It must be initialized before any UI work begins.

```bash
bunx shadcn@latest init
```

During init, choose:
- Style: **Default**
- Base color: your preference (e.g. Slate or Zinc for dark themes)
- CSS variables: **Yes**

This generates `components.json`, `lib/utils.ts` (with `cn`), and wires up Tailwind correctly. All custom components from 21st.dev depend on `cn` from `@/lib/utils` — never skip this step.

## Custom UI Components (from 21st.dev)

These components are defined in `Compontents.md` and placed in `components/ui/`.

### `shader-animation.tsx`
- A full-screen WebGL fragment shader rendered via Three.js.
- **Client component** (`"use client"`).
- Used as the animated background in the **Hero** section.
- No props — self-contained. Handles resize and cleanup automatically.
- Inline `style={{}}` props are intentional here (canvas sizing).

### `glowing-effect.tsx`
- A mouse-proximity-reactive glowing border that wraps card elements.
- **Client component** (`"use client"`). Uses `memo` + `useCallback` for perf.
- Used on **Project cards** and **Skills cards**.
- Place as a direct child inside the card's outer container div (not inside the inner content div).
- Key props: `spread`, `proximity`, `glow`, `disabled`, `borderWidth`, `inactiveZone`.

### `liquid-glass-button.tsx`
- Exports three button components: `Button`, `LiquidButton`, `MetalButton`.
- **Client component** (`"use client"`).
- `Button` — standard shadcn-compatible button with a `cool` variant added.
- `LiquidButton` — glass morphism button with SVG backdrop-filter distortion. Best used on dark/image backgrounds (Hero CTAs).
- `MetalButton` — metallic press-state button with color variants (`default`, `primary`, `success`, `error`, `gold`, `bronze`).
- Uses `@radix-ui/react-slot` for `asChild` support and `cva` for variant logic.

## Site Sections

The home page (`app/page.tsx`) renders these sections in order:

1. **Hero / About Me** — Full-screen intro with `ShaderAnimation` as background, name, title, short bio, and `LiquidButton` CTAs ("View Projects", "Contact Me").
2. **Projects** — Card grid with `GlowingEffect` on each card. Cards show title, description, tech tags, and links.
3. **Skills / Tech Stack** — Visual grid or grouped list, optionally with `GlowingEffect` on skill category cards.
4. **Contact** — Social/email links or a contact form.

Section components live in `components/sections/`.

## Data & Content

All personal content is kept in `data/` as typed TypeScript files — not hardcoded inside components.

- `data/projects.ts` — Array of project objects (`title`, `description`, `tags`, `liveUrl`, `githubUrl`, `featured`)
- `data/skills.ts` — Skills grouped by category (e.g. Languages, Frameworks, Tools)
- `data/profile.ts` — Name, tagline, bio, social links, email

Update these files to change site content without touching component logic.

## Styling Conventions

- Use **Tailwind utility classes** directly in JSX. No separate CSS modules unless unavoidable.
- Follow a **mobile-first** approach — base styles are mobile, `md:` and `lg:` variants layer on top.
- Dark mode is supported via Tailwind's `dark:` variant. Default to dark mode preferred.
- Use CSS custom properties (via shadcn's `globals.css` setup) for design tokens — these power `bg-background`, `text-foreground`, `border-border`, etc.
- Inline `style={{}}` props are allowed only in animation-driven components (e.g. `GlowingEffect`, `ShaderAnimation`).

---

## Component Conventions

- All components are **functional React components** with explicit TypeScript prop types.
- Define prop types with `interface` directly above the component (not in a separate types file unless shared across multiple files).
- Use **named exports** for all components — no default exports except for Next.js page/layout files.
- Mark a component `"use client"` only if it uses hooks, browser APIs, or event listeners. All three 21st.dev components require it.
- Keep components focused: section components handle layout/structure; `ui/` components handle reusable visuals.

---

## Accessibility

- All images must have meaningful `alt` text.
- Interactive elements (buttons, links) must be keyboard-navigable and have visible focus styles.
- Use semantic HTML elements (`<main>`, `<section>`, `<nav>`, `<article>`, `<header>`, `<footer>`).
- Each section should have an `id` attribute matching its nav anchor (e.g. `id="projects"`).
- Color contrast must meet WCAG AA minimum.
- `ShaderAnimation` is decorative — wrap it with `aria-hidden="true"`.

---

## Performance

- Prefer Next.js `<Image>` over `<img>` for all images.
- Keep client components (`"use client"`) minimal — most sections should be React Server Components.
- `ShaderAnimation` is heavy (WebGL). Lazy-load it with `next/dynamic` and `ssr: false` to avoid server-side rendering issues with Three.js.

```tsx
// In Hero.tsx
import dynamic from "next/dynamic"
const ShaderAnimation = dynamic(
  () => import("@/components/ui/shader-animation").then(m => m.ShaderAnimation),
  { ssr: false }
)
```

## Git Conventions

- Branch off `main` for features: `feat/hero-section`, `fix/contact-form`, etc.
- Commit messages follow **Conventional Commits**: `feat:`, `fix:`, `chore:`, `style:`, `refactor:`, `docs:`
- Never commit directly to `main` for significant changes — use a PR.

## Deployment

- Target: **GitHub Pages** (static export) or **Vercel** (recommended for full Next.js support).
- If using GitHub Pages, set `output: 'export'` in `next.config.ts` and configure the base path if the repo is not at the root.
- Environment variables (e.g. contact form API keys) go in `.env.local` and are listed (without values) in `.env.example`.
- Three.js WebGL requires a browser environment — ensure `ShaderAnimation` is loaded with `ssr: false` (see Performance section).
